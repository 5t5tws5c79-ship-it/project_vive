# 큐레이션 게시글 수정·삭제 + 비밀번호 권한 확인 구현 계획

## Context (왜 이 작업을 하는가)

`docs/요구사항.md`의 **필수** 항목 중 "게시글 CRUD(목록/상세/작성/수정/삭제)"와 "비밀번호 권한 확인"이 있다.
커뮤니티 파이프라인 1차 구현(등록/목록/상세/좋아요/댓글, localStorage 기반)은 "최소 침습" 방침으로
수정·삭제를 범위에서 뺐다. `CurationDetailView.vue`의 소유자 박스(비밀번호 입력 + 수정/삭제 버튼)는
현재 `disabled` 상태로 남아있다. 이번 계획은 그 자리를 실제로 채워 필수 항목을 충족시킨다.

## 현재 상태 (참고)

- `src/lib/communityStore.js`: `curations`(반응형 배열, localStorage 영속), `curationById`,
  `addCuration`, `toggleLike`, `addComment`만 존재. 비밀번호 검증/수정/삭제 함수 없음.
- `addCuration`은 이미 `password: password || null` 필드로 저장 중
  (`CurationNewView` 폼에서 입력한 값). `RegisterTrackModal`로 등록한 글은 비밀번호를 안 받으므로
  `password: null` — 이런 글은 원천적으로 수정·삭제 불가로 취급하면 된다(별도 예외 처리 불필요).
- `src/views/CurationDetailView.vue` (약 64~73행): `owner__pw` input, `owner__btn`(수정)/
  `owner__btn--danger`(삭제) 버튼이 모두 `disabled`, 안내 문구
  "플레이스홀더 — 비밀번호 확인과 수정·삭제는 아직 동작하지 않습니다."
- `src/lib/toast.js`의 `showToast(text)`를 성공 피드백에 재사용하면 됨(신규 로직 불필요).

## 변경 파일

### 1. `src/lib/communityStore.js`

기존 함수들과 같은 패턴으로 3개 추가 (뮤테이션 후 기존 `persist()` 호출):

- `verifyPassword(id, password)` — 해당 curation의 `password`가 존재하고 입력값과 일치하면 `true`.
  `password`가 `null`(비밀번호 없이 등록된 글)이면 항상 `false`.
- `updateCuration(id, password, patch)` — `verifyPassword` 통과 시 `{ track, artist, comment }` 등을
  `Object.assign`, `persist()`, `true` 반환. 실패 시 `false`.
- `deleteCuration(id, password)` — `verifyPassword` 통과 시 배열에서 `splice`, `persist()`, `true` 반환.
  실패 시 `false`.

### 2. `src/views/CurationDetailView.vue`

**스크립트**
- `verifyPassword, updateCuration, deleteCuration`을 `communityStore`에서 import, `showToast`를
  `lib/toast`에서 import.
- 상태 추가: `password`(입력값 ref), `pwError`(에러 메시지 ref), `isEditing`(boolean ref),
  `editForm`(ref `{track, artist, comment}` — 수정 모드 진입 시 현재 curation 값으로 초기화).
- `onEdit()`: `verifyPassword(id, password.value)` 확인 → 실패 시 `pwError` 설정 후 return →
  성공 시 `editForm.value`를 현재 값으로 채우고 `isEditing.value = true`, `pwError` 클리어.
- `saveEdit()`: `updateCuration(id, password.value, editForm.value)` → 성공 시
  `isEditing.value = false` + `showToast('수정되었습니다')`.
- `cancelEdit()`: `isEditing.value = false`.
- `onDelete()`: `verifyPassword(id, password.value)` 확인 → 실패 시 `pwError` 설정 → 성공 시
  `deleteCuration(id, password.value)` 호출 후 `showToast('삭제되었습니다')` + `router.push('/community')`.

**템플릿**
- owner 영역의 `disabled` 전부 제거하고 실제 이벤트 연결(`v-model="password"`, `@click="onEdit"`,
  `@click="onDelete"`). 삭제는 파괴적 동작이므로 네이티브 `confirm('정말 삭제하시겠습니까?')` 한 번으로
  즉석 확인(별도 모달까지는 과함, 최소 침습 유지).
- `pwError`가 있으면 기존 힌트 자리에 표시하고, "플레이스홀더 — ..." 문구는 제거.
- `isEditing`일 때 `track`/`artist`/`comment` 표시 영역을 인라인 입력(`v-model="editForm.track"` 등)으로
  스왑 + "저장"/"취소" 버튼. 평소엔 기존처럼 텍스트로만 표시(기존 마크업 유지).
- `curation.password`가 `null`인 글(= `RegisterTrackModal` 경유)은 owner 영역을
  `v-if="curation.password"`로 감싸거나 "비밀번호 없이 등록되어 수정·삭제할 수 없습니다" 안내로 대체.

## 검증

1. `/community/new`에서 비밀번호(4자 이상) 넣고 글 작성 → 상세에서 그 비밀번호로 "수정" 클릭 →
   필드 편집 가능해지는지, "저장" 후 목록/상세에 반영되는지.
2. 틀린 비밀번호로 수정/삭제 시도 → 에러 문구만 뜨고 아무 변화 없는지.
3. "삭제" → confirm → `/community` 목록에서 사라지고 새로고침해도 안 돌아오는지(localStorage 반영 확인).
4. `RegisterTrackModal`로 등록한 글(비밀번호 없음)은 상세에서 수정/삭제 영역이 비활성 안내로 뜨는지.
